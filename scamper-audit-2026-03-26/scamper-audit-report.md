#### SCAMPER: Bi-Weekly Cycle

**Substitute**
The guest-led Show & Tell should be replaced with documented case studies from named, regulated industry peers. A live guest introduces two risks: they may share proprietary methodology, and they may recommend tools not on our approved vendor list. A curated case study from someone in financial services or pharma — with institutional context intact — delivers the same learning with defensible provenance.

The Sprint's framing ("no theory — just hands-on experimentation") must be substituted with "guided application with a validation checkpoint." The behavior is similar; the institutional risk profile is entirely different. I cannot return to my manager and describe my afternoon as "experimentation with no theory."

**Adapt**
The Sprint structure should borrow from clinical training's "observe → practice under supervision → reflect" model. Show a worked example from a constrained environment first. Then apply. The current format inverts this: experiment cold, reflect later. That's the format that produces reckless outputs in compliance-sensitive contexts.

**Modify**
The 45-minute Sprint needs to compress to 25 minutes of structured application — with a decision framework provided upfront that covers: what categories of data are safe to use in AI prompts, what output requires validation before use, and what documentation is recommended. This isn't bureaucracy; it's the terrain I work in.

**Put to Another Use**
The Reflection Pods could double as peer validation documentation. If I can show that I reviewed my outputs with two peers who confirmed my override decisions, that's a legitimate quality assurance step I can reference in a stakeholder conversation. Currently framed purely as personal reflection.

**Eliminate**
The public commitment element of the Tiny Experiment — "before leaving, each participant commits to one small workplace experiment" — has no institutional frame. I'm not in a position to commit on behalf of my organization in a 20-person session. The commitment itself is fine; the public declaration without organizational context creates premature exposure.

**Reverse**
The Sprint should come AFTER a worked example, not before. Participants who see a safe, successful path through constrained territory first will then practice with significantly lower anxiety. The current sequence asks for cold experimentation, which produces defensive behavior, not genuine learning.

#### Marit's 3 Key Improvements

1. **Add a "Constraint Environment Declaration" at cycle start.** Participants declare their operating context (fully regulated / semi-regulated / open) in the first session, and program components include adaptive scaffolding by track. The Sprint for regulated participants includes a data classification step and a validation gate. The Tiny Experiment for regulated participants defaults to the "Validated Application" template. This is not about limiting what people can do — it's about providing a legitimate path through constrained terrain that currently has no roadmap.

2. **Reframe the Tiny Experiment as a "Validated Application" with an optional one-page documentation template.** The template has four fields: use case category, AI tool used, output reviewed (yes/no + by whom), and application status (in use / pending approval / rejected). Completing it is optional. Its existence signals that the program understands that some participants operate in environments where documentation is not bureaucracy — it's the cost of doing anything at all.

3. **Add a fourth Override Log prompt: "What would I tell a stakeholder who asked about this decision?"** This turns a private reflection into a draft professional record. The Override Log's pedagogical power is real — but it stops at the boundary of my organization unless it produces something I can actually use in an institutional conversation. This prompt provides that output without requiring a separate process.

---

#### Marit's 3 Risks

1. **The Sprint's "no theory, just experiment" framing triggers governance anxiety (mechanism: anxiety).** Participants in regulated contexts will not experiment openly in a setting that hasn't established that open experimentation is safe. They will either withdraw from the Sprint mentally (performing safe, trivial tasks that don't expose them) or disengage entirely. The current framing selects for the kind of experimentation that is impossible in the organizations the program claims to serve.

2. **Peer sharing before trust is built (mechanism: comfort).** The Reflection Pods require showing work to strangers with no established context about each other's constraint environments. A comment from an open-sector participant — "why not just try it and see?" — delivered without malice but without awareness of the cost, is enough to trigger institutional anxiety and confirm that this room is not safe. Marit will not return.

3. **No institutional bridge from program to organization (mechanism: identity threat).** The program frames the transformation as "compliant executor → responsible innovator." But it provides no mechanism for crossing that bridge institutionally — no language for explaining the transition to a manager, no stakeholder-facing artifact from the program, no approved path for bringing program learning back into an organizational context. Without this, the transformation stays inside the program and dies the moment Marit returns to her desk.

**SCAMPER: Override Log**

**Modify**
This is the strongest component and the most under-built. Three prompts is not enough. Two additional prompts would transform it:

*"What did you try before you reached this override?"* This question captures the process — the iteration, the failed attempts, the moment of recognition. Without it, the override appears as a single clean decision, which is never what actually happens.

*"What would a less experienced practitioner have done here?"* This question makes craft visible. It names the gap between expertise and novice performance. **Reverse**
The Override Log must be live, not retrospective. If I fill it out after the session, I am reconstructing a decision that happened in the moment. The reconstruction is sanitized — I will produce something coherent and professional-sounding that doesn't capture what actually happened. The real decision was probably messier, faster, and more intuitive than any retrospective account. Build the log into the Sprint itself — not as homework afterward.

**Put to Another Use**
The Override Log could function as a career portfolio document — not a reflection form, but evidence of professional judgment over time. If I complete an Override Log entry for every significant piece of work I do, I have a record of 12 years of decisions I've never been able to articulate before. ## Key Tensions

**Tension 1 — Language adaptation vs. language resistance**
Marit: adapt the program's language to survive institutional contexts.
Yael: resist, because adapting language to bureaucratic mediocrity reproduces it.
*Why it doesn't resolve:* Both positions are internally consistent and serve the person holding them. The program cannot simultaneously signal "experiment freely" and "here is institutional documentation scaffolding." It must choose which participants it's primarily designing for.

**Tension 2 — Override Log purpose (craft portfolio vs. governance document vs. legitimization tool)**
Yael: career document, evidence of professional judgment, craft differentiation.
Daan: governance evidence, organizational adoption argument.
Marit: stakeholder-facing record, survivable in a manager meeting.
*Why it doesn't resolve:* Each purpose implies different prompts, different outputs, and different uses after the session ends. Adding all three purposes to one log either creates incoherence or requires three separate versions of the same component.

**Tension 3 — Who is this program for?**
The program claims to serve all three personas. The debate surfaced that the core learning arc (discover → experiment → reflect → iterate) assumes a linear progression through new capability. Daan is past that arc. Marit is blocked from entering it. Yael is on a different arc entirely (practicing judgment, not discovering capability). The program cannot be the same program for all three without redesigning its progression model.

**Tension 4 — Community promise vs. capability reality**
The program promises peer community. Daan's foundational critique: community promises are only valuable if the room contains people at your level. The program has no stated filtering mechanism. Delivering community requires either accepting a heterogeneous room (which frustrates Daan) or filtering for capability (which creates hierarchy and may reduce diversity of insight).

## Consensus Points

1. **Change the Reflection Pod anchor question from "show the artifact" to "show the moment you overrode AI."** All three personas either proposed this directly or voted YES. The highest-confidence single design change in the audit.

2. **The Override Log is the program's strongest component and is under-built.** All three personas engage with it centrally and all three want it to produce something beyond the session — Marit wants a stakeholder record, Yael wants a craft portfolio prompt, Daan wants a governance argument. The consensus: more prompts, more structural depth, more deliberate output.

3. **The Tiny Experiment commitment structure does not fit all participant types.** All three personas have a different relationship to the commitment: Marit can't commit without institutional permission, Yael wants to commit to a question rather than an experiment, Daan has already done 18 months of experiments and needs a different kind of commitment. Some form of differentiated commitment structure is a high-confidence design signal.

4. **The program's metrics need to shift from quantity to depth.** "80 experiments completed" signals quantity. All three personas are oriented around quality of judgment, depth of redesign, or institutional impact — none of which is captured by a count.

## Decision Framework

When prioritizing changes from this audit, apply the following questions in order:

**1. Does this change serve the Override Log?**
The Override Log is the program's pedagogical spine and the component all three personas engage with most deeply. Changes that strengthen it (additional prompts, live rather than retrospective completion, explicit differentiated purposes) should be prioritized above changes to other components.

**2. Does this change shift the Reflection Pods from output-sharing to process-sharing?**
The Reflection Pod anchor change is the highest-consensus suggestion in the audit. It costs nothing structurally and changes who can meaningfully participate. Implement first, test immediately.

**3. Does this change require deciding who the program is primarily for?**
Marit's constraint tracks, Daan's capability tracks, and Yael's depth tracks are all structurally compatible — but they serve different organizing principles. Before implementing any of them, the program needs to decide: is it organized by constraint environment, capability level, or depth of purpose? These produce different cohort compositions and different participant expectations.

**4. Does this change require differentiating the Tiny Experiment commitment?**
The Legitimacy Track (Daan), Validated Application (Marit), and question-commitment (Yael) are all compatible as optional variants of the same structural slot. The question is whether the program offers variants or defaults. Offering variants without a clear default creates decision paralysis; defaulting without offering variants excludes participants who need something different.

**5. Is this change reversible?**
Changes to framing and language (Override Log prompts, Reflection Pod anchor, metric framing) are low-cost and reversible — test them first. Changes to structure (tracks, sprint format, expo redesign) require more commitment and should be piloted with one cohort before being built into the core model.
